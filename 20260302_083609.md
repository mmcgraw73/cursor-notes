#### Findings — Deep Dive

##### FINDING 1: `fallbackMethod` not wired — 4 fallback methods are dead code

**Severity: LOW (pre-existing — not a regression)**

The `@CircuitBreaker` annotations do not include `fallbackMethod`:
```java
@CircuitBreaker(name = "GetIcpDetailByFileNumberCommand")
// no fallbackMethod = "icpDetailFindByFileNumberFallBack"
```

However, the **original** `@HystrixCommand` annotations also never wired `fallbackMethod`. The old Javadoc even said: *"If graceful degredation [sic] is possible, add `fallbackMethod = "icpDetailFindByFileNumberFallBack"` to the `@HystrixCommand`"* — it was a TODO, never implemented.

The 4 fallback methods (`icpDetailFindByFileNumberFallBack`, `updateIcpDetailFallBack`, `patchIcpDetailFallBack`, `getDiagnosticLegendByCodeFallBack`) have been dead code all along. Updated Javadoc now correctly says to add `fallbackMethod = "..."` to `@CircuitBreaker`. Behavior is unchanged.

To enable in the future:
```java
@CircuitBreaker(name = "GetIcpDetailByFileNumberCommand", fallbackMethod = "icpDetailFindByFileNumberFallBack")
```

##### FINDING 2: `ignoreExceptions` dropped — but safe due to try/catch pattern

**Severity: LOW (safe in practice)**

Old code had:
```java
@HystrixCommand(commandKey = "...",
    ignoreExceptions = { IllegalArgumentException.class, BipException.class, BipRuntimeException.class })
```

New code has no `ignoreExceptions` on `@CircuitBreaker` or in YAML. This looked like a HIGH severity bug initially — if `BipException` (business/validation error) counted as a circuit breaker failure, 20 bad requests could open the circuit and block all subsequent calls including valid ones.

However, examining the actual method bodies:
```java
@CircuitBreaker(name = "GetIcpDetailByFileNumberCommand")
public GetIcpDetailDomainResponse getIcpDetailByFileNumber(...) {
    try {
        response = partnerHelper.icpDetailFindByFileNumber(request);
    } catch (BipException | BipRuntimeException bipException) {
        // CAUGHT HERE — creates a domain response with error message
        domainResponse.addMessage(...);
        return domainResponse;  // returns normally, doesn't throw
    }
}
```

The method **catches** `BipException`/`BipRuntimeException` internally and returns a normal response. The circuit breaker proxy never sees these exceptions — they're swallowed before reaching it. The `ignoreExceptions` in the old Hystrix code was **redundant**. Only uncaught exceptions (e.g., `NullPointerException`, network errors from Feign) would trip the circuit — which is correct behavior.

**Recommendation:** Add `ignoreExceptions` defensively — if someone modifies the catch blocks in the future, the safety net is gone. Not a blocking issue.

##### FINDING 3: `@TimeLimiter` not applied at service method level

**Severity: INFORMATIONAL**

The YAML defines `timelimiter` instances with `timeoutDuration: 20s`, but `@CircuitBreaker` alone does not activate the time limiter. You'd need `@TimeLimiter(name = "...")` on each method for method-level timeouts.

However, the timeout is applied at the **Feign client level** via `spring.cloud.openfeign.circuitbreaker.enabled: true`. Spring Cloud OpenFeign's `CircuitBreakerFactory` (auto-configured by Resilience4j) wraps Feign calls with both the circuit breaker and time limiter when instance names match. The 20s timeout applies where it matters — at the HTTP call boundary.

Old behavior: Hystrix `timeoutInMilliseconds: 20000` wrapped the entire `@HystrixCommand` method. New behavior wraps only the Feign call. Since the service methods do nothing except call `partnerHelper` → Feign, the effective coverage is the same.

##### FINDING 4: Double circuit breaking (informational — pre-existing)

`spring.cloud.openfeign.circuitbreaker.enabled: true` wraps every `DataServiceClient` Feign call in its own circuit breaker. The `@CircuitBreaker` on `IcpApiServiceImpl` wraps the service method that calls `PartnerHelper` → `DataServiceClient`. Each downstream call passes through **two** circuit breakers. This was also true with Hystrix (`feign.hystrix.enabled: true` + `@HystrixCommand`), so behavior is preserved.
